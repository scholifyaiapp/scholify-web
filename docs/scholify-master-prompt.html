SCHOLIFY — FULL PROJECT BRIEF (A–Z)

You are working on Scholify, an AI-tutored ACCA exam-prep web app, live at scholifyapp.com. Read this whole brief before touching product, design, or business decisions — it is the ground truth, assembled from the founder's own docs and verified against the live codebase.

======================================================================
A. IDENTITY & THESIS
======================================================================
- Product: Scholify — "Learn Daily, Grow Steadily." AI persona is Charles: tutor, examiner, coach.
- Who: ACCA candidates, beachhead market Uzbekistan/CIS.
- Problem: ACCA pass rates run 40-55%; CIS tuition centres charge EUR300-800/paper; every existing tool (question banks, video courses) goes silent the moment a student fails — no diagnosis, no path back.
- The pitch, in one line: instead of handing a student 2,000 questions, Scholify says "based on this diagnostic you have a 68% chance of passing FM — here is exactly how we get you to 85%," then walks them there daily.
- Differentiation vs Kaplan/BPP/OpenTuition/ChatGPT-DIY/local players (e.g. Makon AI): pass-probability recalibration tied to real exam outcomes, a recovery flow for retakers, offline-first delivery for patchy connectivity, and CIS-market pricing.
- Caveat for anyone using this brief to raise money: "why now" (falling frontier-model costs) and beachhead market size are FOUNDER ASSERTIONS, not independently sourced — flag them as such, don't repeat them as fact.

======================================================================
B. PRODUCT SURFACE — WHAT'S ACTUALLY SHIPPED
======================================================================
All 15 ACCA papers (Applied Knowledge + Applied Skills + Strategic Professional, incl. options AFM/APM/ATX/AAA) are content-complete and launch-ready. Zero stub routes.

Routes:
  /                     Landing (public, EN/RU)
  /sign-in /sign-up     Auth (Supabase email + Google OAuth)
  /welcome              Onboarding
  /dashboard            Hub: streak, pass probability, today's mission
  /study                Paper hub: chapters, flashcards, AI tutor, mocks
  /study/diagnostic     15-25 question calibration engine, full-syllabus coverage per paper
  /study/analytics      Breakdown by syllabus area, mock trend, knowledge heatmap
  /notes /settings /pricing /support /privacy /terms

Core loop: diagnostic -> pass-probability estimate -> daily plan (deterministic scheduler, not AI-generated -- see section E) -> practice/flashcards/mocks -> mock gate at 60% readiness -> AI Examiner marks written answers -> pass/fail -> recovery ("postmortem") flow if failed, never framed as failure ("you now know exactly where the marks were lost," never "you failed").
Offline-first: entire loop runs with zero API keys against local seed content; every AI/payment path degrades to a clear fallback, never crashes.

C. TECH ARCHITECTURE
======================================================================
- Frontend: Vite + React 18 SPA, TypeScript, Tailwind v4, motion/framer-motion, react-router-dom.
- Hosting: Vercel (Hobby tier — hard 12-serverless-function cap). Backend is deliberately thin: 5 functions total (api/lara.ts, api/stripe.ts, api/paddle.ts, api/social.ts, api/calendar-callback.ts, api/reminders.ts), each multiplexing several operations behind a `?action=` query param dispatcher to stay under the cap. Exceeding 12 functions fails deploys SILENTLY (known historical incident) -- watch this if adding new endpoints.
- Data: Supabase (Postgres + Auth + RLS + Realtime + Storage). Architecture is LOCAL-FIRST: browser localStorage is the primary store, Supabase is a sync/durability layer, not source of truth. Whole-document merge conflict resolution ("busier device wins") -- accepted tradeoff, not a bug.
- Migrations: supabase/migrations/0001..0020, apply in numeric order. 0013 (ai_usage) and 0015 (billing_ai_guardrails: ai_usage_global, ai_rate, subscriptions, study_reminders) are MANDATORY -- AI fails closed without them. 0018 (stripe_events) enables webhook dedup.
- Billing: Stripe is the LIVE rail (Paddle was the original plan, code paths still exist but dormant). Six Stripe env vars required; /api/health returns 503 "half_configured" unless billing is fully configured or not configured at all -- deliberately all-or-nothing.
- Auth/entitlement: plan stored in service-role-only `app_metadata` (JWT-riding, no round-trip cost) with a durable `subscriptions` table as audit trail; AI meter takes the LOWER of the two so neither a stale JWT nor a half-applied webhook over-grants.
- Email: Resend, gated behind RESEND_API_KEY + CRON_SECRET for the reminder cron (fails closed without a secret).
- Testing/CI gates: `npm run build` = typecheck (app + api) -> test -> vite build, in that order -- a failing test or type error blocks the build. Also: `npm run audit:content` (content-floor check per paper), `npm run verify:loading` (every paper's dynamic import actually resolves, catches what tests can't), `npm run validate:chapters`.
- Verified directly (not just claimed): typecheck clean, 191/191 tests passing, 15/15 papers clear the content-launch floor, as of commit 9fa3a83.

======================================================================
D. AI ARCHITECTURE
======================================================================
- Provider selection is env-driven and automatic: api/lara.ts checks ANTHROPIC_API_KEY first, falls back to OPENAI_API_KEY, refuses (fallback mode) if neither is set. No code change needed to switch -- just set the key.
- Current model IDs in code: Anthropic haiku = "claude-haiku-4-5", sonnet = "claude-sonnet-5". OpenAI stopgap = "gpt-4o-mini" (volume tier), "gpt-5.6-terra" (marking/generation tier).
- Model mix by task: Haiku/gpt-4o-mini for high-volume tutor chat; Sonnet/gpt-5.6-terra for AI Examiner marking and content generation (higher-stakes, lower-volume).
- Cost governance (all fail CLOSED, verified in code, not just comments):
    - AI_KILL_SWITCH=1 -- instant global fallback, no redeploy.
    - AI_DAILY_TOKEN_BUDGET (default 5,000,000/day, org-wide) -- once spent, every AI action falls back until midnight UTC.
    - AI_PER_MINUTE_LIMIT (default 8/user) -- atomic via a `bump_ai_rate` RPC so concurrent requests can't both pass.
    - Every DB read in the meter path (`meterAcca()`) explicitly checks the `error` field from supabase-js (which never throws) and DENIES on error -- a missing table or failed read cannot silently read as "0 usage."
- The AI does NOT drive personalization yet: the daily plan is a deterministic scheduler based on live practice accuracy, not the diagnostic's weak-area output, and not LLM-generated. "AI-personalized" in marketing copy currently overstates this -- know the gap before promising more.

E. SECURITY POSTURE (post-audit, verify before trusting blindly)
======================================================================
A founder-run adversarial audit (2026-07-14) found and fixed five serious issues; all five were independently re-verified in the current code (not just the changelog):
  1. Entitlement could not be self-granted (was: any user could write their own plan via user_metadata) -- now service-role-only app_metadata, checked file:line in api/stripe.ts.
  2. AI meter now genuinely fails closed (was: fail-open, because supabase-js never throws so try/catch was dead code).
  3. api/social.ts open email relay removed (was: unauthenticated, could send 100 emails/request from the verified domain).
  4. api/calendar-callback.ts IDOR fixed -- user id now comes only from the verified JWT, never a body-supplied field.
  5. FR paper's 45%-biased-to-option-A answer keys fixed with a real mulberry32 seeded shuffle (a first attempt reusing the existing LCG shuffle FAILED silently -- its low bits don't mix on a 4-element array, it just rotated the bias to a different letter). Wired into every render surface (AccaStudy, AccaDiagnostic, CbeMockRunner), not dead code.
Also closed: Stripe webhook replay protection (stripe_events table, idempotent claim), RLS on every user-data table including calendar_accounts (which stores refresh tokens).
Landing page now explicitly discloses "no users yet, we won't invent testimonials" instead of the fabricated social proof the audit found.
Still open as of last review: rotate the sk_live Stripe key (transited chat once), confirm support@scholifyapp.com actually receives mail, watch the first real (non-sandbox) paid checkout closely.

F. DESIGN SYSTEM (condensed — full bible is a separate document)
======================================================================
- Brand mark: hexagonal circuit of 6 rotated waypoints, MONO/DEEP RED #C80000 primary variant. Wordmark: Plus Jakarta Sans Bold, dotless i with a gradient node.
- Palette: Deep Red #C80000 -> Magenta #E50068 -> Amber #F4A405 gradient (ONE gradient CTA per screen, never on body text). App surfaces run on --sch-* CSS tokens (warm-paper light theme #f7f3f1, dark theme #050508) -- never hardcode hex for surfaces/text.
- Status color (green/amber/red) means ONLY good/at-risk/behind vs a real target -- bandColor(pct, passLine=60). Brand red is identity, never status.
- Type: Plus Jakarta Sans app-wide (h1 26/800, body 14/450, uppercase label 11.5/800). Landing runs a SEPARATE system: Instrument Serif (display) + Geist (sans/mono) + Caveat (handwritten) -- landing is visually pinned, changes are additive only.
- Icons: Lucide only, via a semantic name map -- zero emoji as icons, anywhere in the app.
- Motion: easing [0.22,1,0.36,1], entrances "rise," list stagger 0.05s, hover never shifts layout, max one ambient motion element per screen, prefers-reduced-motion respected globally.
- Voice: "a coach with receipts" -- every big claim quotes the student's own number; never say "failed"; empty states teach what starts the measurement, never fake data.
- Full reference with live token swatches, real icon SVGs, and a standalone paste-ready design prompt: docs/design-bible.html in this repo.

G. CONTENT
======================================================================
- Scale: 15 papers, ~2,400+ original bank questions, 900+ flashcards, 190 rubric-backed written questions (constructed-response papers only -- 4 objective-test-only papers, BT/MA/FA/LW, carry none by design, AI Examiner hidden for those).
- Production pipeline: blueprint -> parallel AI-assisted authoring -> MANDATORY independent re-solve verification -> balanced answer-key positions -> merge. AI may draft/assist but cannot be the final human reviewer (5-stage lifecycle: draft -> source_checked -> calculation_checked -> independently_reviewed -> verified).
- Known process gap (self-caught, not hidden): BT/F1 content was found labeled "verified" on an internal dashboard while actually only draft-stage -- the "verified" label needs active policing, it isn't self-enforcing.
- Recurring cost, not one-time: Examiner Intelligence refreshes every ACCA results week (4x/year x 15 papers); every Finance Act/IFRS change forces re-verification of affected numeric content. Tax content is currently labeled to its Finance Act basis so students know which year they're studying.
- IP posture: invented companies throughout, no reproduced ACCA/Kaplan/BPP material, independence disclaimer present wherever ACCA is referenced.
- Moat honesty: the content pipeline is a documented, reproducible prompt-engineering workflow -- a competent competitor with the same AI tools could rebuild comparable volume in weeks; Kaplan/BPP could match depth in under 6 months if motivated. The verification RIGOR is a real quality edge today, not a structural moat.
